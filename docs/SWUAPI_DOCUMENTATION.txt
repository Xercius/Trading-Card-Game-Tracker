Star Wars: Unlimited – API Specification
=========================================

Document scope
--------------
This file records the base URL, authentication method, rate-limit behaviour, and
full list of known endpoints for the Star Wars: Unlimited card-data API as
observed by the SwuDbImporter (api/Importing/SwuDbImporter.cs) and the existing
reference notes in docs/starwarsunlimitedapi.txt.

IMPORTANT: The endpoints below are internal endpoints operated by Atomic Mass
Games / Fantasy Flight Games for the official card database at
https://starwarsunlimited.com. They are NOT a formally published public API.
They are based on Strapi and may change without notice.


1. BASE URL
-----------
https://admin.starwarsunlimited.com/api/


2. AUTHENTICATION
-----------------
Method  : None (unauthenticated public HTTP GET)
Headers : No Authorization, API-Key, or token header is required.
          The importer sends only the standard headers added automatically
          by .NET's HttpClient (Accept, User-Agent, etc.).
Notes   : Because no credentials are required, all requests must be GET
          requests. Write operations (POST/PUT/DELETE) are not available to
          external consumers.


3. RATE LIMITING
----------------
Official limits : None have been publicly documented by the operator.
Observed limits : No explicit rate-limit response (HTTP 429) has been
                  encountered during normal import operations.
Practical rules applied by the importer:
  - HTTP client timeout : 5 minutes per request
  - Page size cap       : 100 records per page (maximum used in practice)
  - No explicit delay   : Requests are paged sequentially; no artificial
                          sleep is inserted between pages.
Recommendation  : Keep page size at or below 100. If a 429 or 503 response
                  is received, implement exponential back-off starting at 1 s.
                  Avoid issuing more than one concurrent request to this host.


4. ENDPOINTS
------------
All endpoints are relative to the base URL above.
HTTP method for all consumer-accessible endpoints: GET

4.1  card-list
    Full path  : GET https://admin.starwarsunlimited.com/api/card-list
    Purpose    : Returns a paged list of card records (Strapi content-type).
    Used by    : SwuDbImporter.ImportFromRemoteAsync

    Required parameters : none (a bare GET returns all records)
    Common parameters (Strapi-style URL query string):

    Parameter                              Example value      Purpose
    -------------------------------------  -----------------  --------------------------------
    locale                                 en                 Language of returned records
    pagination[page]                       1                  1-based page index
    pagination[pageSize]                   100                Records per page (max ~100)
    sort[0]                                expansion.sortValue:asc  Primary sort
    sort[1]                                cardNumber:asc     Secondary sort

4.2  No other endpoints are currently used by this application.
    The Strapi framework underlying the site likely exposes additional
    content-type endpoints (e.g. /api/expansions, /api/card-types), but they
    have not been verified and are not consumed by any importer in this
    codebase. Any future use of additional endpoints should be documented in
    a new section here before being relied upon in production code.

6. FILTER OPERATOR REFERENCE
------------------------------
Operator      Meaning
-----------   -------------------------------------------
$eq           Equal
$ne           Not equal
$eqi          Equal, case-insensitive
$in           Included in a list
$notIn        Not included in a list
$gt           Greater than
$gte          Greater than or equal
$lt           Less than
$lte          Less than or equal
$contains     Contains substring
$containsi    Contains substring, case-insensitive
$null         Is null (value: true)
$notNull      Is not null (value: true)
$and          All child conditions must match
$or           At least one child condition must match
$not          Negates a condition

Caveat: this is a customised Strapi endpoint. Not every operator is
guaranteed to work for every relational field. Negative filters on
many-to-many relations ($notIn, $ne) can produce unexpected results;
prefer client-side exclusion for those cases.


7. TIMESTAMP METADATA AND UPDATE DETECTION  (Task 2.4 findings)
--------------------------------------------------------------

7.1  Timestamp fields returned by the API
    Every card record in the `attributes` object contains three timestamp
    fields managed automatically by Strapi:

    Field         Type                    Notes
    ------------  ----------------------  ----------------------------------
    createdAt     ISO-8601 UTC string     Set once when the record is first
                                          published; never changes afterwards.
    updatedAt     ISO-8601 UTC string     Bumped by Strapi every time any
                                          attribute on the record is changed.
                                          This is the authoritative field for
                                          detecting modifications.
    publishedAt   ISO-8601 UTC string     Set when the record is published;
                                          may equal createdAt for cards that
                                          go live immediately.

    Example values observed in production data:
      "createdAt":   "2025-08-15T18:29:41.633Z"
      "updatedAt":   "2025-11-10T16:07:21.320Z"
      "publishedAt": "2025-08-15T18:30:00.000Z"

7.2  Date format
    All three fields use ISO 8601 / RFC 3339 format with millisecond
    precision and a literal 'Z' UTC suffix:

      yyyy-MM-ddTHH:mm:ss.sssZ

    When constructing filter query parameters, use the same format:

      filters[updatedAt][$gt]=2025-11-01T00:00:00.000Z

    The C# importer represents these values as `DateTimeOffset?` and
    serialises them back to the same format via `System.Text.Json`.

7.3  Date filtering support
    Date filtering IS supported via the standard Strapi filter syntax.
    Tested operators for timestamp fields:

    Operator   Example                                         Notes
    ---------  ----------------------------------------------  -------------------
    $gt        filters[updatedAt][$gt]=2025-11-01T00:00:00.000Z  strictly after
    $gte       filters[createdAt][$gte]=2025-08-01T00:00:00.000Z on/after date
    $lt        filters[updatedAt][$lt]=2026-01-01T00:00:00.000Z  strictly before
    $lte       filters[createdAt][$lte]=2025-12-31T23:59:59.000Z on/before date

    Combine with other filters using the standard Strapi AND behaviour:

      filters[expansion][code][$eq]=SOR&filters[updatedAt][$gt]=2025-11-01T00:00:00.000Z

7.4  Reliable strategy to detect new/updated cards
    Recommended approach for incremental synchronisation:

    1.  Record the UTC timestamp immediately before each successful import
        run (e.g. `DateTimeOffset.UtcNow` captured at start of run).

    2.  On the next run supply this timestamp as `UpdatedSince` in
        `ImportOptions`.  The importer will add
        `filters[updatedAt][$gt]=<timestamp>` to every API request, so
        only records modified since the previous run are fetched.

    3.  Sort results by `updatedAt:asc` (the importer default when
        `UpdatedSince` is set) so that pages are stable and you can
        resume a partial import by advancing the cursor.

    4.  Apply a small overlap (subtract ≥ 1 minute from the stored
        timestamp) to guard against sub-second clock skew and records
        whose `updatedAt` falls exactly on the boundary.

    5.  The upsert logic in `SwuDbImporter.UpsertAsync` is idempotent:
        re-importing a card that has not changed produces no DB writes
        (no `CardsUpdated` / `PrintingsUpdated` increments), so
        overlapping records are safe to re-fetch.

    6.  Prefer `updatedAt` over `createdAt` as the synchronisation field
        because it catches both newly added cards and corrections to
        existing cards (text, artwork, rarity, etc.).

    Full-expansion fallback: if no stored timestamp is available (first
    run) or if a corruption is suspected, omit `UpdatedSince` and import
    the full expansion.  The importer's upsert behaviour ensures
    idempotency.

    C# `ImportOptions` parameter: `UpdatedSince` (type `DateTimeOffset?`).
    When set, the importer emits `filters[updatedAt][$gt]=...` in the
    query string automatically.
